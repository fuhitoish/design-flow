# Design Flow — 本プラン

**ステータス:** 確定版（v1.1）  
**対象者:** Fuhito（マーケティング部門・グラフィック／ウェブ制作）

## 1. 背景と目的

### 1.1 現状の課題

```
制作開始 → Illustrator で試作
  → 「イメージと違う」「レイアウト変更」「色・構成のズレ」
  → 手戻り・作り直し
```

### 1.2 目指す状態

```
計画段階でトンマナ・構成・配置を完全決定（原稿は段階的に）
  → 制作開始（決定に従うだけ）
  → イメージズレなし、軌道修正なし
```

### 1.3 本プランの3つの役割

1. **プロセスの標準化** — 業界標準のデザインプロセスを、Fuhito が実践できる形に落とし込む
2. **判断の言語化** — 「なぜこの色か」を記録し、ブランド資産として蓄積する（Bellagio / Bellagio Miyabi 等）
3. **システム化の設計図** — Notion または Web アプリ（Phase 2）のデータ構造・画面遷移の根拠とする

## 2. 設計原則

| 原則 | 内容 |
|---|---|
| **コア ＋ モジュール** | 全案件共通の根幹（コア）に、制作物別の専用プロセス（モジュール）を着せ替える |
| **構成・トンマナ優先** | 制作前に「見た目と配置」を決めるのが中心。イメージズレ・レイアウト変更を防ぐ |
| **業界フェーズの可視化** | 各 Step に業界標準プロセス上の位置を明示し、「今どこにいるか」を常に把握する |
| **原稿は段階的** | Step 3 は仮原稿・構成で OK。本番原稿は Step 6 前に確定 |
| **理由の記録** | 色・フォント・レイアウトの各決定に、1〜2文の理由づけを必須とする |
| **遡及を許容** | 矛盾が見つかったら該当 Step に戻る（失敗ではなく正常なフロー） |
| **既存ブランドの例外** | ガイドラインがあるブランドは、コア Step 4（ビジュアル・ファウンデーション）を適用・スキップ選択可 |

### 2.1 最重要関門（Fuhito 向け）

| 優先度 | 関門 | 内容 |
|---|---|---|
| **★★★** | Step 4 トンマナ | 色・フォント・ビジュアル方向・グリッドを確定 |
| **★★★** | Step 5 レイアウト | 台割・WF・配置。仮原稿でレイアウト実証 |
| **★★★** | 最終ゲート（§7） | 「完全に見えている状態」の確認 |
| **★★** | Step 3 コンセプト | キーメッセージ・方向性・構成・仮原稿 |
| **★** | 本番原稿ロック（§6.1） | Step 6 直前。本番テキストへの差し替え |

## 3. 二層構造：Design Flow × 業界標準フェーズ

Design Flow は **実務チェックリスト**、業界フェーズは **地図** です。両方を並記して使う。

### 3.1 対応表

| Design Flow | 業界標準フェーズ | Double Diamond | この段階で決めること |
|---|---|---|---|
| **Step 1** | Brief / 要件定義 | Discover | 誰に・何のため・制約 |
| **Step 2** | Research / 戦略 | Discover | 競合・トレンド・差別化 |
| **Step 3** | Concept / コンテンツ設計 | Define | メッセージ・方向性・構成・仮原稿 |
| **Step 4** | Visual Design / Art Direction | Define → Develop | 色・フォント・ビジュアル・グリッド |
| **Step 5** | Information Design / レイアウト | Develop | 台割・WF・導線・配置 |
| **（§6.1）** | Copy Finalization | Develop → Deliver | 本番原稿への差し替え |
| **Step 6** | Production / 実装 | Deliver | Illustrator 等で制作・納品 |

### 3.2 ステータス記録の例

```
現在の Step     : Step 5（モジュール A）
業界フェーズ    : Information Design / Wireframe
Diamond         : Develop
次のアクション  : P2-3 見開き WF 完了
```

## 4. アーキテクチャ概要

```
┌─────────────────────────────────────────────────────────────┐
│  コア（全案件共通）                                          │
│  Step 1 要件 → Step 2 リサーチ → Step 3 コンセプト・コンテンツ │
│              → Step 4 ビジュアル・ファウンデーション         │
└──────────────────────────┬──────────────────────────────────┘
                           │ 制作物種別を選択（Step 1 で決定）
         ┌─────────────────┼─────────────────┬─────────────┐
         ▼                 ▼                 ▼             ▼
   モジュールA         モジュールB       モジュールC    モジュールD
   印刷物・パンフ      Web・LP          ポスター・単一広告  バナー・SNS
         │                 │                 │             │
         └─────────────────┴─────────────────┴─────────────┘
                           ▼
                  本番原稿ロック（§6.1）
                           ▼
                    Step 6 制作・実装
                  （決定に従うのみ。構造変更は不可）
```

## 5. コア・フロー（Step 1〜4）

制作物の種類に関わらず、**制作に入る前に骨格をロック** する根幹プロセス。

### Step 1: プロジェクト要件（目的とターゲット）

**業界フェーズ:** Brief / 要件定義  
**目的:** 「何のため・誰に・どんな制約で作るか」を明確にする。

- [ ] プロジェクト名・制作背景・最終ゴール（1〜2文）
- [ ] ターゲット層の明確化（ペルソナ：年代・職業・課題・価値観）
- [ ] **制作物の種類の選択**（後続モジュールを決定）
- [ ] 媒体・仕様の初期確定（サイズ・ページ数／CMS・対応デバイス等。未確定は「要確認」と記録）
- [ ] 必須掲載要素のリストアップと優先順位（1〜3）
- [ ] 制約・ルール（使わない色・避けるトーン・参考にしない競合）
- [ ] ビジュアル参考画像 3〜5 枚

**ゲート:** 上記すべてチェック → Step 2 へ

---

### Step 2: リサーチ・戦略分析

**業界フェーズ:** Research / 戦略  
**目的:** ターゲット・競合・トレンドを理解し、表現の土台を作る。

- [ ] ペルソナの深堀り（Step 1 より詳しい理解を 3〜5 点抽出）
- [ ] 競合・参考企業 5〜10 社のビジュアル分析（色・フォント・トーン・レイアウト）
- [ ] 「我々がやってはいけないこと」の明文化
- [ ] トレンド調査（取り入れる／無視するの判断含む）
- [ ] インスピレーション画像 20〜30 枚以上

**ゲート:** 上記すべてチェック → Step 3 へ

---

### Step 3: コンセプト・コンテンツ設計

**業界フェーズ:** Concept / コンテンツ設計  
**目的:** メッセージ・ビジュアル方向・情報構成を確定する。原稿は **仮文（字数・段落構造が本番想定）** で OK。

- [ ] キーメッセージの策定（1〜2文。以降すべての判断基準）
- [ ] ビジュアル方向性の決定（形容詞 2〜3 個 + ムードボード 3〜5 枚）
- [ ] キーワード 3〜5 個の抽出
- [ ] 必須掲載要素ごとの **仮原稿**（copy-draft.txt：字数・段落・見出し有無を本番想定）
- [ ] 支給画像・必須ロゴなど素材の確保（未入手は「入手方法・期限」を記録）
- [ ] セクション別の文字数見積もり

**ゲート:** キーメッセージ・方向性・構成・仮原稿（字数確定）が揃っている → Step 4 へ

> 本番原稿（copy-master.txt）は Step 6 前（§6.1）に確定する。

---

### Step 4: ビジュアル・ファウンデーション（基本トンマナ）

**業界フェーズ:** Visual Design / Art Direction  
**目的:** 色・フォント・画像方向・グリッド・余白を確定し、モジュール段階以降は変更しない。

> **コア最重要関門。** 既存ブランドガイドラインがある場合は適用し、該当項目をスキップ可。

#### 4-1. カラーパレット

- [ ] メイン・サブ（2〜3）・アクセント・ニュートラルの決定
- [ ] RGB（Web）/ CMYK（印刷）両値の記載
- [ ] 各色の理由づけ（1〜2文）
- [ ] Illustrator 等での色見本作成・確認

#### 4-2. タイポグラフィ

- [ ] 見出し・本文フォントの決定（Google Fonts 等）
- [ ] サイズ階層（h1 / h2 / h3 / 本文 / キャプション）
- [ ] 和文・洋文・数字混在の表示確認

#### 4-3. 画像・ビジュアル方向

- [ ] 画像タイプ（写真 / イラスト / グラフィック）の決定
- [ ] 色・雰囲気・トーンの言語化
- [ ] 参考画像 5〜10 枚

#### 4-4. グリッド・余白

- [ ] グリッドシステム（印刷: マージン・段組 / Web: 列数・ブレークポイント・ガター）
- [ ] ホワイトスペースの戦略（1〜2段落で言語化）

**ゲート:** Step 3 のキーメッセージ・キーワードと整合し、曖昧な項目がゼロ → モジュールへ

## 6. 着せ替えモジュール（Step 5）

**業界フェーズ:** Information Design / レイアウト（Wireframe）  
**目的:** 制作物固有の構造・配置を確定する。**モジュール最重要関門。**

Step 1 で選択した制作物種別に応じ、**1 モジュールのみ** を適用する。  
WF 検証には Step 3 の **仮原稿** を使用する。

### モジュール A：印刷物・パンフレット

- [ ] **仕様確定:** 仕上がりサイズ、ページ数、用紙、折り、加工、部数、納期
- [ ] **台割設計:** 全ページの役割と、どこにどの原稿が入るかの割り当て
- [ ] **ワイヤーフレーム（紙面）:** 実寸グリッドでの配置（仮原稿でレイアウト実証）
- [ ] **画像スペース指定:** サイズ・アスペクト比・トリミング方向
- [ ] **適用確認:** Step 4 の色・フォント・グリッドがレイアウトで機能するか
- [ ] **入稿前確認:** CMYK 変換、塗り足し、トンボ

### モジュール B：Web サイト・LP

- [ ] **仕様確定:** ブレイクポイント（PC / Tablet / SP）、CMS、動作環境、主要機能
- [ ] **情報設計:** サイトマップ（階層）と導線設計
- [ ] **ワイヤーフレーム（画面）:** 各ブレークポイントでの要素配置・並び替え
- [ ] **インタラクション設計:** ホバー・クリック・メニュー開閉・フォーム仕様
- [ ] **CMS 仕様:** 動的ページ・テンプレート・カスタムフィールド（該当時）
- [ ] **適用確認:** Step 4 の色・フォント・グリッドが各ブレークポイントで機能するか

### モジュール C：ポスター・単一広告

- [ ] **仕様確定:** 掲出場所、サイズ、縦横比、印刷色数
- [ ] **環境シミュレーション:** 遠距離視認性、周囲色とのコントラスト
- [ ] **視線誘導設計:** Z の法則等の適用、メインコピー配置のロック
- [ ] **ワイヤーフレーム:** 実寸での要素配置（仮原稿でレイアウト実証）

### モジュール D：バナー・SNS 画像

- [ ] **仕様確定:** 各プラットフォームの規定サイズ、ファイル容量上限
- [ ] **セーフゾーン確認:** UI（アイコン・テキスト）が被らないエリアの特定
- [ ] **要素の断捨離:** 仮原稿から、媒体に合わせてテキストを極限まで削る
- [ ] **ワイヤーフレーム:** 媒体別レイアウト（仮原稿でレイアウト実証）

## 7. 本番原稿ロック（Step 6 前）

**業界フェーズ:** Copy Finalization  
**目的:** 制作に入る直前に、仮原稿を本番原稿（copy-master.txt）に差し替える。

- [ ] 全セクションの本番テキストが確定（クライアント承認含む）
- [ ] 文字数・段落構造が Step 5 WF と整合（変更があれば WF 再確認）
- [ ] 素材が最新版に更新されている

**ゲート:** 上記すべてチェック → 最終ゲート（§8）へ

> 小規模案件・文案変更なしの場合は、Step 3 仮原稿 = 本番としてスキップ可（status-tracker に記録）。

---

## 8. 制作・実装（Step 6）

**業界フェーズ:** Production / 実装  
**目的:** Step 5 で確定した設計と本番原稿に従い、Illustrator / InDesign / HTML・CSS 等で実装する。

### やること

- 確定レイアウト・本番原稿・素材の配置
- Step 4 のカラー・タイポグラフィの適用
- 細部調整（字詰め、色微調整、折り・ノンブル等）
- 納品ファイルの作成（PDF / 入稿データ / 公開）

### やらないこと

- 大幅なレイアウト変更
- 色・フォントの根本的な変更
- ページ／セクション構成の変更

→ 上記が必要になった場合は **Step 5（または該当コア Step）の詰め不足** と判断し、制作を止めて遡及する。

---

## 9. 最終ゲート：「完全に見えている状態」か？

**業界フェーズ:** Pre-production Sign-off  
モジュール完了 + 本番原稿ロック後、制作（Step 6）に進む前に以下をすべて YES にする。

- [ ] キーメッセージを自分で述べられる
- [ ] ターゲット層について具体的に話せる
- [ ] 各色・フォントについて理由を述べられる
- [ ] ページ／セクションごとの「役割」を説明できる
- [ ] 仕様変更の質問に、設計理由で答えられる
- [ ] 実装を「ほぼ想像」できる（制作開始時の迷いがない）

**1 つでも NO → 該当 Step に遡及**

## 10. システム化（Phase 2）

### 10.1 エンティティ

```
Project（親）
├─ コア: Step 1〜4 の情報・チェック状態
├─ industry_phase: brief | research | concept | visual | layout | copy_final | production
├─ module_type: pamphlet | web | poster | banner
├─ Module（子）: 選択したモジュールの Step 5 チェックリスト
├─ Documents: ムードボード、カラーパレット、タイポ、WF、copy-draft、copy-master
└─ status: core_1 | core_2 | ... | module | copy_final | gate | production | done
```

### 10.2 画面遷移イメージ

```
プロジェクト作成
  → 制作物種別を選択（モジュール決定）
  → コア Step 1〜4 を順次入力（業界フェーズ表示付き）
  → モジュール専用項目を入力
  → 本番原稿ロック
  → 最終ゲート確認
  → Step 6（制作）ステータスへ
```

### 10.3 プロトタイプ実装の優先順位

| Phase | 内容 | 成果物 |
|---|---|---|
| **Phase 1** | 本プランに基づくチェックリストの手動運用 | テンプレ + サンプル案件 |
| **Phase 2a** | コア ＋ 1 モジュール（パンフ or Web）の Web プロトタイプ | このリポジトリ |
| **Phase 2b** | 全モジュール対応・Documents 添付・ブランド資産の横展開 | 本番 Web アプリ |
