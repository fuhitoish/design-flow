# モジュール型デザインワークフロー設計書

## 1. システムコンセプト：「コア ＋ 着せ替えモジュール」
すべての制作物に共通する「根幹の意思決定（コア）」をベースとし、制作物の種類に応じて必要な「専用の決定事項（モジュール）」をアドオンしてフローを完成させる。

*   **コア（全案件共通）：** 要件、原稿確定、基本トーン（色・フォント）
*   **モジュール（制作物別）：** パンフレット用、Web用、バナー用などの専用プロセス

---

## 2. コア・フロー（すべての案件で必ず通る根幹）
制作物の種類に関わらず、一切の作業を始める前に「完全にロック」する基本事項。

### Step 1: プロジェクト要件（目的とターゲット）
- [ ] プロジェクト名と最終ゴール（誰に・どうなってほしいか）
- [ ] ターゲット層の明確化（ペルソナ）
- [ ] 制作物の種類（ここで後続の「モジュール」を選択）

### Step 2: コンテンツの完全ロック（原稿と素材）
※ダミーテキストでのデザイン進行を禁止し、手戻りを防ぐ最大の関門。
- [ ] 必須掲載要素のリストアップと優先順位
- [ ] 実原稿（テキストデータ）の確定
- [ ] 支給画像・必須ロゴなどの素材確保

### Step 3: ビジュアル・ファウンデーション（基本トンマナ）
※既存ブランドの場合は、既存のルールを適用してスキップ可能。
- [ ] カラーパレット（メイン・サブ・アクセント）の決定と理由づけ
- [ ] タイポグラフィ（見出し・本文のフォント指定）
- [ ] ビジュアルの方向性（写真のトーン、余白の持たせ方）

---

## 3. 着せ替えモジュール（制作物別の拡張フロー）
Step 1で選択した「制作物の種類」に応じて、以下の専用プロセスがフローに合流する。

### モジュールA：【印刷物・パンフレット】
物理的な制約と、ページものの情報設計に特化したプロセス。
- [ ] **仕様確定：** 仕上がりサイズ、ページ数、用紙、加工（折り等）
- [ ] **台割設計：** 全ページの役割と、どこにどの原稿が入るかの割り当て
- [ ] **ワイヤーフレーム（紙面）：** 実寸グリッドでのテキストボックス配置（原稿が収まるかの実証）
- [ ] **実装・入稿前確認：** CMYK変換、塗り足し、トンボの確認

### モジュールB：【Webサイト・LP】
画面サイズの可変性と、ユーザーの操作（インタラクション）に特化したプロセス。
- [ ] **仕様確定：** ブレイクポイント（PC/SP等）、使用CMS、動作環境
- [ ] **情報設計：** サイトマップ（階層）と導線設計
- [ ] **ワイヤーフレーム（画面）：** 各ブレイクポイントにおける要素の並び替え・挙動定義
- [ ] **インタラクション設計：** ホバー時、クリック時、メニュー開閉時の挙動定義

### モジュールC：【ポスター・単一広告】
一瞬の視認性と、設置環境に特化したプロセス。
- [ ] **仕様確定：** 掲出場所、サイズ、縦横比
- [ ] **環境シミュレーション：** 遠距離からの視認性、周囲の色とのコントラスト確認
- [ ] **視線誘導設計：** Zの法則等の適用、メインコピーの配置ロック

### モジュールD：【バナー・SNS画像】
狭い面積での情報伝達と、媒体ごとの制約に特化したプロセス。
- [ ] **仕様確定：** 各プラットフォームの規定サイズ、ファイル容量上限
- [ ] **セーフゾーン確認：** UI（アイコンやテキスト）が被らないエリアの特定
- [ ] **要素の断捨離：** 狭小スペースに合わせ、Step 2の原稿からさらに極限までテキストを削る

---

## 4. システム化（Webアプリ）に向けたデータ構造のイメージ
この設計は、リレーショナルデータベースで非常に効率的に管理できます。

*   **Project（親モデル）:**
    *   Step 1〜3のコア情報を持つ。
*   **Module（子モデル）:**
    *   プロジェクトが「パンフレット」なら `PamphletModule` を紐付け。
    *   プロジェクトが「Web」なら `WebModule` を紐付け。

これにより、UI上では「プロジェクト作成 ＞ Webを選択 ＞ コア入力 ＞ Web専用項目入力」というスムーズな画面遷移とチェックリスト生成が実現します。
